# Shakespeare download site

The public landing page and macOS download for Shakespeare. The entire visible
site is one quiet, generated photograph of a sea-facing writing desk, with a
single download action. A tiny Cloudflare Worker handles redirects and assets.

## Local development

```bash
npm ci
npm run dev
```

The local site is served at `http://localhost:3000`.

## Build and verify

```bash
npm run build
npm test
```

There is no client JavaScript. Wrangler serves `public/` directly, and retired
routes such as `/how-it-works` return to the home page.

## Release downloads

The public archive is generated by the signed and notarized release workflow.
Local `make package` output is ad-hoc signed for development and must not be
published. Versioned archives live in the `shakespeare-releases` R2 bucket;
the Worker resolves this stable path through an atomic manifest:

```bash
/downloads/Shakespeare-latest.zip
```

The checksum response is generated from the same manifest as the ZIP and the
Worker fails closed if the manifest or archive is missing. Downloads are not
part of static site deployments, so changing the landing page cannot erase or
silently replace a release.

## Publish

```bash
npx wrangler deploy --config wrangler.jsonc
```

Production: <https://writeshakespeare.com>

Cloudflare Workers Builds should use `Website` as the root directory, `npm ci && npm run build` as the build command, `npx wrangler deploy --config wrangler.jsonc` as the deploy command, and `main` as the production branch. GitHub Actions is not used.
